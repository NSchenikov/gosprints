Спринт 7: Реализация ETL-пайплайна с Kafka

Задача:
Добавить в систему обработку потоковых данных с помощью Kafka. Будем строить ETL (Extract-Transform-Load) пайплайн, который позволяет обрабатывать события о задачах в реальном времени.
Функционал:

    Extract (извлечение данных):
        Вместо обычных HTTP-запросов или gRPC-обращений задачи теперь отправляют события в Kafka при каждом значимом изменении (created, updated, completed).
        Например, после обновления статуса задачи (status: in_progress → completed) сервис записывает событие в Kafka-топик task_events.

    Transform (обработка данных):
        Отдельный сервис-потребитель (ETL Worker) читает события из Kafka.
        Производит нужные преобразования (например, фильтрует задачи, логирует статистику, готовит данные для аналитики).
        Валидация и фильтрация входных событий перед записью в хранилище.

    Load (загрузка в хранилище):
        Записываем обработанные данные в аналитическую базу данных (PostgreSQL, ClickHouse, MongoDB — на выбор).
        Опционально: развернуть ClickHouse для быстрого анализа потоковых данных.

Пример событий Kafka (JSON-пейлоад):

Топик task_events:

{
"task_id": "123",
"user_id": "42",
"event_type": "task_completed",
"timestamp": "2025-02-01T12:00:00Z"
}

Топик task_analytics (данные после обработки):

{
"user_id": "42",
"tasks_completed": 10,
"avg_completion_time": 120
}

Дополнительные возможности (если хватит времени):

    Разбить события на партиции Kafka по user_id, чтобы упростить горизонтальное масштабирование.
    Реализовать retry-механику (если база данных не доступна, сервис должен повторить попытку загрузки данных).

Как интегрируется с остальной системой:

Task Service → Kafka (публикация событий при изменении задач).
ETL Worker (читает из Kafka, обрабатывает данные и пишет в БД).

Предполагаемая архитектура
Task Service → Kafka (task-events) → ETL Worker → ClickHouse/PostgreSQL (аналитика)
......................↘
......................Notification Service (WebSocket)
