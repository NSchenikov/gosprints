Спринт 7: Реализация ETL-пайплайна с Kafka

Задача:
Добавить в систему обработку потоковых данных с помощью Kafka. Будем строить ETL (Extract-Transform-Load) пайплайн, который позволяет обрабатывать события о задачах в реальном времени.
Функционал:

    Extract (извлечение данных):
        Вместо обычных HTTP-запросов или gRPC-обращений задачи теперь отправляют события в Kafka при каждом значимом изменении (created, updated, completed).
        Например, после обновления статуса задачи (status: in_progress → completed) сервис записывает событие в Kafka-топик task_events.

    Transform (обработка данных):
        Отдельный сервис-потребитель (ETL Worker) читает события из Kafka.
        Производит нужные преобразования (например, фильтрует задачи, логирует статистику, готовит данные для аналитики).
        Валидация и фильтрация входных событий перед записью в хранилище.

    Load (загрузка в хранилище):
        Записываем обработанные данные в аналитическую базу данных (PostgreSQL, ClickHouse, MongoDB — на выбор).
        Опционально: развернуть ClickHouse для быстрого анализа потоковых данных.

Пример событий Kafka (JSON-пейлоад):

Топик task_events:

{
"task_id": "123",
"user_id": "42",
"event_type": "task_completed",
"timestamp": "2025-02-01T12:00:00Z"
}

Топик task_analytics (данные после обработки):

{
"user_id": "42",
"tasks_completed": 10,
"avg_completion_time": 120
}

Дополнительные возможности (если хватит времени):

    Разбить события на партиции Kafka по user_id, чтобы упростить горизонтальное масштабирование.
    Реализовать retry-механику (если база данных не доступна, сервис должен повторить попытку загрузки данных).

Как интегрируется с остальной системой:

Task Service → Kafka (публикация событий при изменении задач).
ETL Worker (читает из Kafka, обрабатывает данные и пишет в БД).

Предполагаемая архитектура
Task Service → Kafka (task-events) → ETL Worker → ClickHouse/PostgreSQL (аналитика)
......................↘
......................Notification Service (WebSocket)

Потом я понял что нет уведомлений ws о создании новой задачи. В терминале notification-service запустился, но нет логов о подключении к Kafka. Переменные окружения не передаются или Kafka не может подключиться. Прологировал notification-service/internal/kafka/consumer.go. Начал проверять что происходит когда создаю задачу; поставил логи в task-service/internal/kafka/producer.go. Проверял вызовы функции PublishTaskEvent внутри продюсера - и этих логов не появляется, значит функция не вызывается. Перед публикацией ивента поставил лог о том что ивент должен опубликоваться но этот лог в терминале не появляется. Лог не появился в выводе, значит что HTTP-запрос на создание задачи не доходит до этого обработчика. Возможно проблема в том, что api-gateway проксирует запросы в task-service через gRPC, а не через HTTP. Ваш TaskHandler в task-service не используется для HTTP-запросов, потому что task-service не имеет HTTP-сервера. Правильная архитектура

Клиент -> api-gateway (HTTP) -> task-service (gRPC) -> repository
......................................\
...................................... -> Kafka (событие)

События должны публиковаться внутри gRPC методов task-service, а не в HTTP хендлерах.
Добавил публикацию событий в grpc-сервер в task-service/internal/grpc/task/server/server.go, вызвав PublishTaskEvent. В main task-service обновил создание продюсера kafka с топиком, и при запуске gRPC сервера передал этот продюсер. Отправляю задачу и получаю Write error: context canceled - контекст запроса был отменён до того, как Kafka успел записать сообщение, возможно потому, что gRPC метод CreateTask завершается и отменяет контекст, а я пытаюсь писать в Kafka в том же контексте. Написал отдельный контекст для Kafka в task-service/internal/grpc/task/server/server.go, отдельный контекст с таймаутом внутри task-service/internal/kafka/producer.go внутри функции PublishTaskEvent.
